{How Frontier AI Models Treat Information Sources in AI-for-Software-Engineering Questions}

\author{Ajoy Dey}
\affiliation{%
  \institution{Bangladesh University of Engineering and Technology}
  \country{Bangladesh}}
\email{}

\begin{abstract}
AI-assisted software engineering is now discussed through many kinds of
sources: software engineering conferences, programming-language venues, arXiv
papers, benchmarks, company blogs, documentation, GitHub repositories, and
practitioner reports. This project studies how frontier and open-weight AI
models treat those sources when asked questions about AI in software
engineering. I designed a reproducible prompt-based study with a 30-venue
conference baseline, 10 prompts, 30 raw model responses, parsers for rankings
and citations, and analysis scripts for ranking agreement, source-type
distribution, domain concentration, and link validation. The collected
responses came from OpenRouter free routes for Gemma 4 31B, GPT-OSS 120B, and
Nemotron 3 Ultra. The pipeline extracted 205 ranking rows, 86 citation rows,
and 86 source links. Results show that models usually rank ICSE, FSE, ASE,
ISSTA, and MSR highly for AI-for-software-engineering, but source-heavy prompts
draw heavily from arXiv and web-accessible resources. The study also finds
practical limitations: free API routes were rate-limited, source links include
dead or unverifiable URLs, and model rankings depend strongly on prompt depth
and source-pressure instructions. The repository contains code, data, raw
responses, parsed tables, figures, and this report so the study can be
reproduced and extended.
\end{abstract}

\section{Introduction}

Modern software developers often solve problems by moving between academic and
industrial sources. In my daily software work with AI, I regularly search
academic papers for durable research ideas and industrial resources for
practical implementation details, APIs, benchmarks, and deployment guidance.
This made the project personally meaningful: if AI assistants are becoming
part of software engineering work, then it is important to understand whether
they value established research sources, whether they overuse recent web
material, and whether they cite exact sources reliably.

This report responds to the project request to explore idiosyncrasies across
frontier models in the treatment of information sources for AI in software
engineering. The project has two main goals:

\begin{enumerate}
  \item study model understanding of source impact by asking models to rank
  conferences and venues related to AI-assisted software engineering; and
  \item study retrieval and attribution behavior by asking models for exact
  links to papers, blogs, repositories, documentation, benchmarks, and other
  internet sources.
\end{enumerate}

The final artifact is not only this report. It is a reproducible research
package containing prompts, raw responses, parsing scripts, analysis scripts,
figures, tables, and limitations.

\section{Research Questions}

The study was organized around seven research questions:

\begin{description}
  \item[RQ1:] Which conferences do AI models consider most important for AI in
  software engineering?
  \item[RQ2:] Do model-generated conference rankings correlate with a baseline
  derived from CORE-style ranks and CSRankings areas?
  \item[RQ3:] Do models prefer academic papers, company blogs, documentation,
  arXiv papers, GitHub repositories, news articles, or general web pages?
  \item[RQ4:] Do models cite sources from their own organization or ecosystem
  more often than competing sources?
  \item[RQ5:] Do models over-prioritize recent LLM and code-generation work
  compared with older software engineering research?
  \item[RQ6:] Do shallow prompts produce more web/blog sources while deeper
  prompts produce more academic citations?
  \item[RQ7:] Which responses contain dead, unverifiable, ambiguous, or likely
  hallucinated source metadata?
\end{description}

\section{Dataset Design}

\subsection{Conference Baseline}

The first dataset is a 30-venue master conference list stored in
\texttt{data/conferences/master\_conference\_list.csv}. The list covers:

\begin{itemize}
  \item software engineering venues: ICSE, FSE, ASE, ISSTA, MSR, ICSME, SANER,
  RE, MODELS, ICPC, ICST, COMPSAC, and SAC;
  \item programming-language and formal-methods venues: PLDI, OOPSLA, POPL,
  ECOOP, CAV, TACAS, FM, and VMCAI; and
  \item adjacent AI, data, web, and HCI venues: NeurIPS, ICML, ICLR, AAAI,
  IJCAI, KDD, WWW, CHI, and UIST.
\end{itemize}

The columns are \texttt{conference}, \texttt{acronym}, \texttt{full\_name},
\texttt{field}, \texttt{core\_rank}, \texttt{csrankings\_area}, and
\texttt{baseline\_rank}. I intentionally included lower-rank or more
specialized venues such as COMPSAC, ICST, and SAC. This tests whether models
only repeat the most prestigious venues or can also recognize specialized
sources that may be useful for practical AI-for-SE questions.

CORE and CSRankings are used as comparison signals, not as absolute truth.
CORE focuses on venue ranking categories, while CSRankings is area-based and
publication-count oriented~\cite{coreportal,csrankings}. Therefore, the
baseline rank is a pragmatic comparison instrument rather than an objective
measure of research quality.

\subsection{Prompt Battery}

The prompt set contains 10 prompts stored in
\texttt{data/prompts/prompt\_set\_v1.md}, with prompt metadata in
\texttt{data/prompts/prompt\_metadata.csv}. The prompts vary by depth and
source pressure:

\begin{itemize}
  \item conference ranking prompts;
  \item sourced conference ranking prompts;
  \item influential-paper prompts;
  \item exact-source disclosure prompts;
  \item recent-work prompts;
  \item area-mapping prompts;
  \item practitioner-guidance prompts;
  \item hallucination-stress prompts;
  \item lower-rank venue prompts; and
  \item source-validation audit prompts.
\end{itemize}

Most prompts request JSON with \texttt{ranked\_items},
\texttt{references}, and \texttt{uncertainty\_notes}. This structure makes the
responses easier to parse while still allowing models to express uncertainty.

\section{Model Collection Method}

\subsection{Models}

The final collection used OpenRouter free routes for three
models~\cite{openrouter}:

\begin{itemize}
  \item \texttt{google/gemma-4-31b-it:free}
  \item \texttt{openai/gpt-oss-120b:free}
  \item \texttt{nvidia/nemotron-3-ultra-550b-a55b:free}
\end{itemize}

The collection script is \texttt{src/query\_models.py}. It saves every raw
response as JSON under \texttt{data/raw\_responses/}. Each file includes the
provider, model, prompt id, run id, timestamp, temperature, system prompt,
user prompt, API parameters, and exact response text.

\subsection{Collection Parameters}

The target decoding setting was temperature 0. Because the free model routes
were rate-limited, some calls used smaller \texttt{max\_tokens} limits to avoid
timeouts. The final raw dataset contains 30 real API response files. The
composition is shown in Table~\ref{tab:collection}.

repeated Nemotron runs. This produces a real 30-response dataset and also
allows a small stability check across repeated runs.

\section{Parsing and Analysis Pipeline}

The pipeline is designed to be reproducible from the repository root.

\begin{verbatim}
python src/run_pipeline.py
python src/validate_links.py \
  --input data/parsed/source_links.csv \
  --output data/parsed/source_links_validated.csv
\end{verbatim}

\subsection{Parsing}

\texttt{src/parse\_responses.py} extracts three tables:

\begin{itemize}
  \item conference rankings:
  \texttt{data/parsed/conference\_rankings.csv};
  \item citations:
  \texttt{data/parsed/citations.csv}; and
  \item source links:
  \texttt{data/parsed/source\_links.csv}.
\end{itemize}

The parser first attempts JSON parsing. If the response is malformed or
wrapped in markdown fences, it uses a lenient JSON extraction fallback. It also
extracts URLs, DOIs, and arXiv identifiers with regular expressions when a
model does not provide clean structured output.

\subsection{Ranking Metrics}

\texttt{src/analyze\_rankings.py} joins model-ranked venues against the
conference baseline. It computes:

\begin{itemize}
  \item Spearman correlation with baseline rank;
  \item Kendall tau with baseline rank;
  \item Spearman correlation with numeric CORE score;
  \item top-5 overlap with the baseline; and
  \item mean absolute rank difference.
\end{itemize}

\subsection{Source Metrics}

\texttt{src/analyze\_sources.py} measures source-type distribution, domain
concentration, citation overlap using Jaccard similarity, publication-year
distribution, and hallucinated-citation candidates. Source types include
\texttt{academic\_paper}, \texttt{conference\_page}, \texttt{arxiv},
\texttt{company\_blog}, \texttt{documentation},
\texttt{github\_repository}, \texttt{news},
\texttt{benchmark\_or\_dataset}, and \texttt{unknown}.

\texttt{src/validate\_links.py} performs automated URL validation. A link is
marked \texttt{valid}, \texttt{dead\_link}, \texttt{unverifiable}, or
\texttt{ambiguous}. This is not a perfect hallucination detector, but it is a
useful first pass for identifying suspicious source claims.

\section{Results}

\subsection{Parsed Dataset Size}

The final pipeline run extracted:

\begin{itemize}
  \item 205 ranking rows;
  \item 86 citation rows;
  \item 86 source-link rows; and
  \item 86 validated source-link rows.
\end{itemize}

These counts show that the prompt design was able to elicit both ranked venue
judgments and exact source disclosures.

\subsection{RQ1 and RQ2: Conference Ranking Behavior}

The ranking prompts generally placed ICSE, FSE, ASE, ISSTA, and MSR near the
top. This is reasonable because these venues are central to software
engineering, automated software engineering, software testing, and mining
software repositories. AI-specific venues such as ICLR, NeurIPS, and ICML also
appeared, especially when models connected AI-for-SE to LLM and code-model
research.

Table~\ref{tab:rankingmetrics} reports selected ranking metrics. The strongest
agreement came from GPT-OSS on the sourced ranking prompt and Nemotron on the
lower-rank venue prompt. The lower-rank prompt improved coverage of specialized
software engineering venues and produced a top-5 overlap of 1.00 for Nemotron
\texttt{final02}.

\subsection{RQ4: Domain Concentration}

The most common domain was \texttt{arxiv.org}. For Gemma, the top extracted
domains included \texttt{arxiv.org}, \texttt{github.com},
\texttt{csrankings.org}, \texttt{portal.core.edu.au}, \texttt{owasp.org}, and
\texttt{huggingface.co}. For Nemotron, the strongest concentration was also
\texttt{arxiv.org}, followed by domains such as \texttt{conf.researchr.org},
\texttt{docs.github.com}, \texttt{doi.org}, \texttt{github.blog}, and
\texttt{github.com}. This pattern suggests that models used broadly available
academic and developer-facing web sources more than closed publisher pages.

The evidence for self-organization preference is limited in this dataset.
There are some GitHub and GitHub documentation links, but the collected models
did not overwhelmingly cite their own provider ecosystems. A larger balanced
dataset would be needed to make a strong claim about organizational bias.

\subsection{RQ5 and RQ6: Recency and Prompt Sensitivity}

Recent LLM and code-generation work appeared frequently. Examples include
Codex-style code generation papers~\cite{chen2021evaluating}, SWE-bench
style software engineering agent benchmarks~\cite{jimenez2023swebench}, and
other arXiv-accessible work. This indicates a recency pull toward LLM-era
software engineering research. However, older and established software
engineering venues still appeared in ranking prompts, especially ICSE, FSE,
ASE, ISSTA, and MSR.

Prompt sensitivity was visible. Ranking prompts emphasized conferences and
prestige. Exact-source prompts produced many arXiv URLs, GitHub links, and
web-accessible pages. The lower-rank venue prompt changed the answer surface:
it encouraged inclusion of specialized venues that may otherwise be omitted.
This supports the idea that shallow prompts can reinforce common top-venue
answers, while deeper prompts with explicit source pressure expose richer and
more diverse source choices.

\subsection{RQ7: Link Validity and Hallucination Risk}

Automated validation of 86 extracted links produced:

\begin{itemize}
  \item 72 valid links;
  \item 12 dead links; and
  \item 2 unverifiable links.
\end{itemize}

Dead and unverifiable links are important because they may indicate
hallucinated URLs, changed URLs, temporary access failures, or websites that
block automated validation. I therefore treat validation as a warning signal
rather than final proof of hallucination. Manual checking is still needed for
high-stakes claims.

\section{Discussion}

\subsection{What I Learned}

This project showed me that AI assistants do not simply retrieve neutral
source lists. Their answers are shaped by prompt wording, source pressure, API
behavior, and the model's learned associations between a topic and well-known
venues. When asked broadly, models tend to name obvious high-impact venues.
When asked for exact sources, they often move toward accessible links such as
arXiv, GitHub, documentation, and conference pages.

This matters for software engineering practice. In real software work, an
engineer often needs both academic depth and industrial immediacy. Academic
papers help with foundations and evidence; documentation, repositories, and
company blogs help with implementation. A good AI assistant should help users
move across both worlds, while clearly separating verified sources from
uncertain or generated metadata.

\subsection{Threats to Validity}

\textbf{API availability.} The strongest threat is the OpenRouter free-route
rate limit. Gemma and GPT-OSS could not be collected for all 10 prompts during
the experiment window. I did not fabricate missing responses. Instead, I
documented the limitation and collected repeated Nemotron runs.

\textbf{Baseline validity.} CORE and CSRankings are useful signals but they
measure different things. A venue can be specialized and useful even if it is
not ranked at the top of a general prestige baseline.

\textbf{Parsing validity.} The parser handles structured JSON and fallback
text extraction, but model responses are messy. Some venue names or citations
may be missed or normalized imperfectly.

\textbf{Link validation.} Automated link validation can fail because of
temporary network problems, redirects, bot blocking, publisher restrictions,
or changed URLs. Manual checking remains necessary.

\textbf{Model versioning.} Free API routes may change underlying models or
providers over time. Raw responses and timestamps are stored to preserve the
actual observed data.

\section{Conclusion}

This project designed and executed a reproducible study of how AI models treat
information sources in AI-for-software-engineering questions. The results show
that models recognize the centrality of major software engineering venues such
as ICSE, FSE, ASE, ISSTA, and MSR, but source-heavy prompts shift responses
toward arXiv and other easily linkable web sources. The lower-rank venue prompt
was useful because it encouraged models to include specialized venues that
matter for practical software engineering research but may be overlooked in
shallow prompts.

The project also exposed practical research challenges. Free API routes can be
rate-limited, model output formats can be inconsistent, and exact URLs can be
dead or unverifiable. These are not just inconveniences; they are part of the
phenomenon being studied. If AI systems are used as research assistants or
software engineering assistants, their source behavior must be measured,
validated, and documented.

Overall, this study provides a working first version of a reproducible
benchmark for source treatment in AI-for-software-engineering answers. Future
work should collect balanced runs across more models, add Crossref or OpenAlex
metadata validation, and compare API responses with web-interface responses
from chat tools used by everyday software engineers.
